## Why calibrate before gating

`cortex_review` (CXEG-04) and the CXEG-07 consistency lens exist to catch
structural elegance regressions and style drift — but any automated
"taste" signal risks becoming a taste-gate that blocks PRs on a reviewer's
mood rather than on anything that actually predicts trouble. Both engines
are advisory-only by construction (`CortexConfig.elegance_advisory_only`,
default `true`; the consistency lens structurally cannot flip a
`review_run` verdict — see `src/review/consistency.rs`'s module doc), but
"advisory" still means someone reads the flag and reacts to it, so a high
false-positive rate against real merged history erodes trust fast.

Calibration answers one question empirically, before any threshold is
trusted: **of the PRs that actually merged and shipped fine, how many would
this scoring have flagged?** That rate is a proxy false-positive rate. A
flag on a PR that shipped and was fine is (usually) a false positive; a flag
on a PR that had to be reverted or hotfixed shortly after is (usually) a
true positive that the excludable revert/hotfix filter deliberately keeps
out of this specific proxy (see "Exclusions" below) — the corpus this
harness replays is "PRs that were treated as acceptable," and a rushed
hotfix is exactly the case where that assumption doesn't hold.

## Methodology

1. **Corpus.** Fetch the last N `state: closed` PRs for a configured
   `--owner/--repo` via the git-private forge tool (S9 — see
   `src/bin/cortex_calibrate.rs`'s module doc), filter to `merged: true`,
   and stop once N merged PRs are collected or the page-count safety cap is
   hit.
2. **Diff resolution.** For each merged PR, resolve its changed-file list
   from `base.sha`/`head.sha` via the forge's `CommitsCompareDiff` endpoint.
   A PR whose compare response carries no recognizable per-file list is
   flagged `diff_unavailable: true` and excluded from the scored sample —
   see the "Known limitation" note in `README.md`'s Calibration section for
   why this can happen and what it means if it happens a lot.
3. **Scoring, dry/capture-only.** Each resolved diff is scored with:
   - `cortex::review::compute_review` (CXEG-04) — the exact function
     `cortex_review` itself calls — for a `risk_score`/`band`/
     `risk_signals`.
   - `review::run_consistency_lens_dry` (CXEG-07) — a narrow wrapper around
     the same `consistency::maybe_run` `review_run` calls, with
     `CortexConfig.enable_tier_c` forced on for the replay regardless of the
     live environment's setting. **Never** calls `review::maybe_record_findings`
     — nothing from a calibration run reaches the KGFIND findings store.
4. **Exclusions.** A PR whose title/body looks like a revert or hotfix
   (`cortex::calibrate::looks_like_revert_or_hotfix`) is, by default,
   dropped from the SCORED sample (`--include-reverts` to keep it in) — it
   is always counted in the corpus total either way. This keeps a rushed
   emergency fix's noise from skewing the FP rate downward-looks-fine or
   upward-looks-bad in either direction; a revert/hotfix is neither a clean
   "this pattern is fine" signal nor cleanly scoreable against the "it
   shipped and was fine" assumption the rest of the corpus relies on.
5. **Would-have-flagged.** A scored PR "would have been flagged" if its
   structural `band` is `elevated`/`high`, OR the consistency lens raised at
   least one finding (`cortex::calibrate::PrRecord::would_flag`). A `low` or
   `unknown` band with zero consistency findings is not a flag.
6. **Rate + recommendation.** `false_positive_rate = would_have_flagged /
   scored_prs`, compared against `--target-fp-rate` (default 10%). Below
   `--min-sample` (default 20) scored PRs, the report is flagged
   `sample_small: true` and declines to recommend a threshold change — a
   small sample can't reliably distinguish a real problem from noise. At or
   above the minimum, the recommendation either says "no change needed"
   (rate at or below target) or names the single highest-firing signal
   (structural or consistency-lens category) as the first thing to retune,
   pointing at the specific env var that controls it
   (`CORTEX_RISK_WEIGHT_*`, `CORTEX_TIER_B_PERCENTILE`,
   `CORTEX_RISK_BAND_ELEVATED_CUT`, `CORTEX_RISK_SCORE_THRESHOLD`).

All of the above (steps 5-6) is pure, unit-tested math in
`src/cortex/calibrate.rs` (`compute_fp_rate`) — independent of any live
Gitea corpus or network access; `cargo test -p terminus-rs` exercises it
against hand-checked fixtures. Steps 1-4 are the (network-touching) harness
in `src/bin/cortex_calibrate.rs`.

## Report format

A generated report (this file, after a live run) contains:

- **Summary** — total PRs examined, scored count, excluded-revert count,
  diff-unavailable count, would-have-flagged count, the false-positive rate
  vs. target, and the `sample_small` flag.
- **Per-signal firing rate** — a table of every structural signal kind
  (`centrality_spike`, `complexity_spike`, `fan_out_explosion`,
  `community_boundary_crossing`, `semantic_duplication`) and every
  consistency-lens finding category that fired at least once, each with its
  fired count, sample size, and rate, ranked highest-first.
- **Recommendation** — the plain-language guidance described in step 6
  above.
- **Raw report (JSON)** — the same numbers as machine-readable JSON
  (`cortex::calibrate::report_to_json`), for feeding into further tooling.
- **Replayed PRs** — the full corpus (number, title, merged, band,
  revert/hotfix flag, diff-unavailable flag) for manual spot-checking.

## Re-running after a threshold change

Calibration is meant to be iterative: adjust the env var(s) the
recommendation names, re-run `cortex_calibrate` against the same (or a
fresher) corpus, and confirm the false-positive rate moved toward the
target before considering `CORTEX_ENABLE_TIER_C=true` (or tightening
`CORTEX_RISK_SCORE_THRESHOLD`/`CORTEX_RISK_BAND_ELEVATED_CUT`) for a live
deployment.
